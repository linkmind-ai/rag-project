## 핵심 로직 #1: Double-Checked Locking 패턴

### 파일 위치
- `apps/stores/vector_store.py:64-82` (ElasticsearchStore.initialize)
- `apps/graphs/rag_graph.py` (RAGGraph.initialize)

### 핵심 개념
멀티코루틴 환경에서 싱글톤 초기화의 안전성과 성능을 동시에 보장하는 패턴입니다.

```
┌─────────────────────────────────────────────────┐
│            Double-Checked Locking               │
├─────────────────────────────────────────────────┤
│ 1. 첫 번째 체크 (락 없이)                        │
│    └─ 이미 초기화됨? → 즉시 반환 (빠른 경로)     │
│                                                 │
│ 2. 락 획득                                      │
│    └─ 동시 초기화 시도 방지                     │
│                                                 │
│ 3. 두 번째 체크 (락 안에서)                      │
│    └─ 대기 중 다른 코루틴이 초기화했을 수 있음   │
└─────────────────────────────────────────────────┘
```

## 핵심 로직 #2: 하이브리드 검색 스코어 정규화

### 파일 위치
- `apps/stores/vector_store.py:252-330` (hybrid_search 메서드)

### 핵심 개념
벡터 검색과 BM25 키워드 검색의 스코어 범위가 다르므로 정규화가 필요합니다.

```
┌─────────────────────────────────────────────────┐
│         스코어 정규화 필요성                     │
├─────────────────────────────────────────────────┤
│ • 벡터 검색 (코사인 유사도): 0.0 ~ 1.0          │
│ • BM25 키워드 검색: 0.0 ~ ∞ (가변적)            │
│                                                 │
│ → 두 스코어를 직접 합산하면 BM25가 지배적       │
│ → Min-Max 정규화로 0~1 범위로 통일              │
└─────────────────────────────────────────────────┘

공식: final_score = (vector × 0.5) + (keyword × 0.5)
```

### 리뷰 포인트
- [ ] `max(scores)`가 0일 때 ZeroDivisionError 처리가 되어 있는가?
- [ ] 가중치(vector_weight)를 0.7로 바꾸면 어떤 효과가 있는가?
- [ ] 두 검색 결과에 모두 있는 문서는 어떻게 처리되는가?

## 핵심 로직 #3: N3 하이브리드 근거 식별

### 파일 위치
- `apps/graphs/rag_graph.py:257-338` (_identify_evidence_node 메서드)

### 핵심 개념
LLM의 추론 능력과 키워드 매칭의 정확성을 결합하여 답변 근거를 식별합니다.

```
┌─────────────────────────────────────────────────┐
│          N3 하이브리드 근거 식별 로직            │
├─────────────────────────────────────────────────┤
│ 1단계: 키워드 기반 후보 추출                     │
│   └─ 답변과 문서 간 키워드 일치도 계산 (≥10%)   │
│                                                 │
│ 2단계: LLM 기반 근거 식별                        │
│   └─ 프롬프트로 LLM에게 인덱스 요청             │
│                                                 │
│ 3단계: 하이브리드 결합                           │
│   ├─ 일치도 ≥20%: 보강 (LLM이 놓친 경우)        │
│   └─ 일치도 <5%: 제거 (LLM 환각 방지)           │
└─────────────────────────────────────────────────┘
```
